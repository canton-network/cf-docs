The Ledger API is the primary interface between your application and a Canton participant node. It provides gRPC and JSON endpoints for submitting commands, reading the transaction stream, and managing parties and packages.

## API Versions

Canton 3.x uses Ledger API v2. The v2 API is the current stable interface. The earlier v1 API (from Canton 2.x) is no longer supported.

## Core Services

### Command Submission

- **CommandService** — Submit commands and wait for completion in a single call. Recommended for most use cases.
- **CommandSubmissionService** — Submit commands asynchronously without waiting for completion. Use with CompletionService for advanced control flow.

### Transaction Stream

- **UpdateService** — Subscribe to a stream of ledger updates (transaction trees, flat transactions, reassignments). This is the primary way to read ledger data.
- **StateService** — Query the current active contract set and get the current ledger end.

### Party and Package Management

- **PartyManagementService** — Allocate new parties, list existing parties
- **PackageService** — Upload DAR packages, list uploaded packages

### Identity and Metadata

- **VersionService** — Query the API version and supported features
- **UserManagementService** — Manage ledger API users and their rights

### Interactive Submission (External Signing)

- **InteractiveSubmissionService** — Prepare transactions for external signing. Used by the Wallet SDK and exchange integrations for external party workflows.

## Access Methods

### gRPC

The native Ledger API uses gRPC (HTTP/2 + Protocol Buffers). Connect on the participant's Ledger API port (default: 5001).

Proto definitions are available in the [Canton repository](https://github.com/digital-asset/canton/tree/main/community/ledger-api/src/main/protobuf/com/daml/ledger/api/v2).

### JSON API

The JSON API provides an HTTP/REST interface that wraps the gRPC Ledger API. In Canton 3.x, it is integrated into the participant node and configured via the `http-ledger-api` section. Access it on the JSON API port (default: 7575).

The JSON API is the recommended access method for TypeScript/JavaScript applications and web frontends. The OpenAPI specification for a running JSON API instance is available at `/v2/docs/openapi` on the JSON API port.

## Authentication

All Ledger API calls require a valid JWT token. The token must include:

- **`sub` claim** — The user ID (for user-scoped operations)
- **`aud` claim** — The expected audience for your deployment

Tokens are obtained from the OIDC provider configured for your validator. See [Wallet Configuration](/docs-main/integrations/wallet/configuration) for environment-specific setup.

## Daml Code Generation

Use an [OpenAPI binding generator](https://github.com/OpenAPITools/openapi-generator) for the language of your choice if you are using the JSON API.  Use [`protoc`](https://protobuf.dev/programming-guides/proto3/) to generate language bindings from protobufs.  Use DPM to generate type-safe Java bindings from your Daml models:

```bash
# Generate Java bindings
dpm codegen-java

# Generate JavaScript/TypeScript bindings
dpm codegen-js
```

Generated bindings provide type-safe wrappers for creating contracts, exercising choices, and parsing transaction events.
