The JSON API provides an HTTP/REST interface to the Canton Ledger API. It runs as a sidecar process alongside a Canton participant and translates JSON HTTP requests into gRPC Ledger API calls.

## Quick Reference

| Property | Value |
|----------|-------|
| **Protocol** | HTTP/1.1 + JSON |
| **Default port** | 7575 |
| **Underlying API** | Ledger API v2 (gRPC) |
| **Authentication** | JWT Bearer tokens |

## When to Use the JSON API

The JSON API is the recommended access method when:

- You're building browser-based frontends or web applications
- Your language/framework works better with HTTP/JSON than gRPC
- You want simpler tooling for development and debugging (curl, Postman, etc.)
- You're using the TypeScript/JavaScript Wallet SDK

Use gRPC directly when:

- You need maximum performance for high-throughput streaming
- You're using Java/Scala and want the native gRPC experience
- You need access to gRPC-specific features not exposed via JSON

## Key Endpoints

### Version Check

```bash
curl http://localhost:7575/v2/version
```

### Command Submission

Submit a command via HTTP POST with a JSON body specifying the command details, workflow ID, and party information.

### Active Contract Queries

Query the active contract set for contracts matching specific templates and filter criteria.

### Transaction Streaming

The JSON API supports WebSocket connections for streaming transaction updates, equivalent to the gRPC UpdateService.

## Configuration

The JSON API connects to a participant's gRPC Ledger API port. In cn-quickstart LocalNet deployments, it is pre-configured and available at `http://localhost:7575`.

For production deployments, the JSON API is typically deployed as a sidecar container in the same pod or service as the participant.

## Authentication

Pass a JWT Bearer token in the `Authorization` header:

```bash
curl -H "Authorization: Bearer <token>" http://localhost:7575/v2/...
```

The token must be valid for the OIDC provider configured for your deployment.
