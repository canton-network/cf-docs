The Canton Console is an interactive Scala REPL that connects to Canton nodes for administration, debugging, and automation. It provides direct access to participant, sequencer, and mediator operations through a typed API.

For full operational documentation, see the [Global Synchronizer section](/docs-main/global-synchronizer/understand/introduction), which covers node setup and management in detail. This page provides a brief orientation to the Console as a development and operations tool.

## Overview

The Console is bundled with the Canton distribution. When you start a Canton process, the Console is available as an interactive shell. You can also connect a remote Console to a running node.

# Getting console access to Canton nodes

For more involved debugging and disaster recovery, direct access to the console of a Canton node (participant, sequencer, mediator) might be required. Steps to obtain such access:

Requirements:

- Direct access to the Canton node process
- Canton binary

Once you see the following banner for the console you have successfully gained access

``` text
_____            _
/ ____|          | |
| |     __ _ _ __ | |_ ___  _ __
| |    / _` | '_ \| __/ _ \| '_ \
| |___| (_| | | | | || (_) | | | |
\_____\__,_|_| |_|\__\___/|_| |_|

Welcome to Canton!
```

## Participant console

1.  Obtain an authentication token as specified in [the Canton authentication docs](https://docs.digitalasset.com/operate/3.4/howtos/secure/apis/jwt.html)

2.  Ensure you can access the participant's ports 5001 and 5002

3.  Add the configuration to a local file <span class="title-ref">console.conf</span>
    ```
    canton {
      remote-participants {
        participant {
          admin-api {
            port = 5002
            address = localhost
          }
          ledger-api {
            port = 5001
            address = localhost
          }
          token = "<auth token>"
        }
      }
      features.enable-preview-commands = yes
      features.enable-testing-commands = yes
      features.enable-repair-commands = yes
    }
    ```

4.  Run the docker command

    > <div class="parsed-literal">
    >
    > docker run -it --rm --network host -v \$(pwd)/console.conf:/app/app.conf /canton: --console
    >
    > </div>
    >
    > <Warning>
>
    > <div class="title">
    >
    > Important
    >
    >
</Warning>
    >
    > If you run the participant using the docker compose setup the docker command must be run with the docker network used by the participant. Adjust the configuration to connect to the participant container:
    >
    > ```
    > canton {
    >   remote-participants {
    >     participant {
    >       admin-api {
    >         port = 5002
    >         address = participant
    >       }
    >       ledger-api {
    >         port = 5001
    >         address = participant
    >       }
    >       token = "<auth token>"
    >     }
    >   }
    >   features.enable-preview-commands = yes
    >   features.enable-testing-commands = yes
    >   features.enable-repair-commands = yes
    > }
    > ```
    >
    > Running docker with the default network (<span class="title-ref">splice-validator</span>):
    >
    > <div class="parsed-literal">
    >
    > </div>
    >
    > docker run -it --rm --network splice-validator -v \$(pwd)/console.conf:/app/app.conf /canton: --console
    >
    > </div>

## Access in a K8s cluster

In a K8s cluster you can use a debug pod to access the console directly from the cluster.

First you can create a pod running the right canton version using:

```bash
kubectl debug "${POD_NAME}" --image "$(kubectl get pod "${POD_NAME}" -o json | jq -re '.spec.containers[0].image')" -i -t -- bash
```

where <span class="title-ref">POD_NAME</span> is the name of the participant/sequencer/mediator pod.

Once you are inside the running pod you can install a text editor and create the config file <span class="title-ref">console.conf</span> that is described above.

```bash
$ apt-get update
$ apt-get install -y vim
$ vim console.conf # paste in the config from above
$ /app/bin/canton -v -c console.conf
```

What you can do with the Console:

- **Party management** -- Create parties, list hosted parties, manage party-to-participant mappings
- **Package management** -- Upload DAR files, inspect deployed packages, check package dependencies
- **Synchronizer connections** -- Connect participants to synchronizers, list connections, manage reconnection
- **Topology management** -- View and modify the topology state (key rotations, namespace delegations, party allocations)
- **Health monitoring** -- Check node status, view active connections, inspect processing queues
- **Debugging** -- Inspect transaction trees, check conflict states, trace command processing

## Quick Examples

```scala
// List all parties hosted on this participant
participant.parties.list()

// Upload a DAR package
participant.dars.upload("/path/to/my-project.dar")

// Check participant health
participant.health.status

// List connected synchronizers
participant.synchronizers.list_connected()

// Allocate a new party
participant.parties.enable("Alice")
```

## Node References

The Console organizes operations by node type. Each node type exposes a different set of commands:

**Validator (participant)** -- The most commonly used reference. The console object is named `participant`. Handles party management, package uploads, ledger operations, and synchronizer connections.

```scala
participant.parties.list()
participant.dars.upload(darPath)
participant.synchronizers.connect_local(synchronizer)
```

**Sequencer node** -- Manages message ordering for a synchronizer.

```scala
sequencer.health.status
```

**Mediator node** -- Handles transaction confirmation for a synchronizer.

```scala
mediator.health.status
```

## Running the Console

### Embedded Console

When you start Canton with a configuration file, the Console starts automatically:

```bash
canton --config my-config.conf
```

The Console prompt appears after the node finishes initializing. Node references (`participant`, `sequencer`, `mediator`) are pre-bound based on your configuration.

### Remote Console

You can connect a Console to a node running elsewhere:

```bash
canton remote-console --host localhost --port 2902
```

This is useful for managing nodes in staging or production environments without SSH access to the node machine.

## Scripting

The Console supports loading Scala scripts for repeatable operations:

```scala
// In the Console
load("/path/to/my-script.sc")
```

Scripts have access to the same node references and APIs as the interactive Console. Common uses include:

- Automated party setup for test environments
- Topology configuration for multi-node deployments
- Health check routines that run periodically
- Data export scripts for auditing

See [Scala bindings](/docs-main/sdks-tools/language-bindings/scala) for more on writing automation scripts.

## Tab Completion

The Console provides tab completion for discovering available commands. Type a node reference followed by a period, then press Tab:

```
participant.<TAB>
  dars    synchronizers    health    id    keys    parties    topology    ...
```

This is the fastest way to explore the API without referring to documentation.
