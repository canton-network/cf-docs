Scala is the implementation language of Canton itself. While most application developers use Java or TypeScript bindings, Scala access is available through the Canton Console and Canton's internal APIs. This makes Scala the right choice for advanced automation, custom health checks, and operational scripting.

## Canton Console and Scala

The Canton Console is a Scala REPL (read-eval-print loop) that connects directly to Canton nodes. Every command you run in the Console is Scala code executed against Canton's internal APIs.

```scala
// In Canton Console: list active parties on a participant
participant.parties.list()

// Upload a DAR package
participant.dars.upload("/path/to/my-project.dar")

// Check node health
participant.health.status
```

Because the Console is a full Scala environment, you can write functions, import libraries, and build complex automation scripts without leaving the REPL. See [Canton Console](/docs-main/sdks-tools/cli-tools/canton-console) for more on using the Console interactively.

## When to Use Scala

Scala bindings are a good fit for:

- **Operational automation** -- Scripts that manage node topology, rotate keys, or configure synchronizer connections
- **Custom monitoring** -- Health check scripts that query node state and emit metrics
- **Advanced testing** -- Integration tests that need direct access to Canton internals (e.g., simulating network partitions or clock manipulation)
- **Console extensions** -- Reusable functions loaded into the Console at startup for your team's common operations

For standard application backends, Java (via `dpm codegen-java`) or TypeScript (via `dpm codegen-js`) are better choices. They have first-class code generation support and are designed for Ledger API interaction.

## Scala API Access

Canton's Scala APIs are organized around node references. In the Canton Console, you interact with nodes through typed objects:

```scala
// Participant operations
participant.domains.list_connected()
participant.topology.party_to_participant_mappings.list()

// Sequencer operations (if managing a synchronizer)
sequencer.health.status

// Mediator operations
mediator.health.status
```

These APIs expose the full surface of Canton's Admin API in a type-safe Scala interface. The tab-completion in the Console helps you discover available methods.

## Writing Automation Scripts

You can write Scala scripts as `.sc` files and execute them in the Canton Console:

```scala
// backup-topology.sc
val parties = participant.parties.list()
val mappings = participant.topology.party_to_participant_mappings.list()

println(s"Found ${parties.size} parties")
println(s"Found ${mappings.size} topology mappings")

// Export to file or external system
```

Load and run the script in the Console:

```scala
// In Canton Console
load("/path/to/backup-topology.sc")
```

## Build Configuration

If you are writing a standalone Scala project that uses Canton's APIs (outside of the Console), you need the Canton libraries on your classpath. Add them as dependencies in your `build.sbt`:

```scala
libraryDependencies += "com.digitalasset.canton" %% "canton-community-base" % cantonVersion
```

Check the [Canton 3.x documentation](https://docs.digitalasset.com) for the current artifact coordinates and version numbers.

## Reference Documentation

The Scaladoc for Canton's Scala APIs is published with each Canton release. It covers the Console command set, node reference types, and the topology management API. Refer to the [Canton 3.x SDK documentation](https://docs.digitalasset.com) for the latest Scaladoc.
