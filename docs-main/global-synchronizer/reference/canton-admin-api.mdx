The Canton Admin API provides gRPC-based administrative access to participant, sequencer, and mediator nodes. In the context of the Global Synchronizer, it is used for node management, topology inspection, and operational tasks.

## Participant Admin API

The participant Admin API (default port: 5002) covers:

- **Health and status** — Node health, connected synchronizers, activity status
- **Topology management** — Party-to-participant mappings, package vetting, namespace delegations, synchronizer parameters
- **Synchronizer connectivity** — Connect, disconnect, and reconnect to synchronizers
- **Package management** — Upload DARs, list packages, manage vetting
- **Key management** — Generate, list, and rotate cryptographic keys
- **Pruning** — Manage ledger data retention
- **Repair** — ACS import/export for disaster recovery

For detailed usage examples, see [Essential Commands](/global-synchronizer/canton-console/essential-commands).

## Sequencer Admin API

The sequencer Admin API (default port: 5009) covers:

- **Health and status** — Sequencer operational status, connected members
- **Pruning** — Manage sequencer data retention

## Mediator Admin API

The mediator Admin API (default port: 5007) covers:

- **Health and status** — Mediator operational status
- **Pruning** — Manage mediator data retention

## Access Methods

### Canton Console

The Canton Console wraps Admin API calls in a Scala REPL. This is the recommended way to interact with the Admin API for debugging and operational tasks. See [Console Overview](/global-synchronizer/canton-console/console-overview).

### Direct gRPC

For programmatic access, connect directly to the gRPC Admin API ports. Proto definitions are available in the Canton repository.

### Kubernetes Debug Pod

In Kubernetes deployments, access the Admin API via a debug pod. See [Console Overview](/global-synchronizer/canton-console/console-overview) for the K8s access procedure.

## Authentication

The Admin API is typically accessed within a trusted network (VPN, private network, or K8s cluster). When tokens are required, use the same OIDC provider configured for your deployment.
