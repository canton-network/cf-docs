The Canton SDK (also called the dApp SDK) is the development toolchain you use to build, test, and deploy applications on Canton Network. It bundles the Daml compiler, code generation tools, a local sandbox environment, and the `dpm` package manager into a single installable distribution.

## What's included

The SDK ships as a versioned release managed through `dpm`. Each release includes:

- **Daml compiler** (`dpm damlc`) -- compiles Daml smart contracts into DAR files (Daml Archives)
- **dpm** -- the command-line package manager that replaces the deprecated `daml` assistant
- **Code generation** -- produces type-safe bindings. Java codegen (`dpm codegen-java`) is first-class with full feature parity. TypeScript/JavaScript bindings (`dpm codegen-js`) are also available.
- **Sandbox** (`dpm sandbox`) -- runs a full Canton node in a single process for local development
- **Daml Script** (`dpm script`) -- executes scripts against the ledger for testing and automation
- **Daml Studio** (`dpm studio`) -- launches the VS Code extension with syntax highlighting, type checking, and error diagnostics
- **PQS** (`dpm pqs`) -- the Participant Query Store, which projects ledger data into SQL tables. Supports SQL joins for rich queries across contract types and retains historical data for audits.
- **Daml Shell** (`dpm daml-shell`) -- interactive client for PQS

Alpha-stage code generation for Scala (`dpm codegen-alpha-scala`) and a new TypeScript codegen (`dpm codegen-alpha-typescript`) are also available.

# Digital Asset Package Manager (Dpm)

`dpm` is a command-line tool that does a lot of useful things related to the SDK. It is a **drop-in replacement** for the now deprecated `Daml Assistant<daml-assistant>`.

## Pre-requisites

Dpm currently runs on Windows, macOS and Linux.

For full functionality, you must have installed:

1)  [vscode download](https://code.visualstudio.com/download)
2)  JDK 17 or greater, installed and part of your <span class="title-ref">JAVA_HOME</span>. If you do not already have a JDK installed, try OpenJDK or [Eclipse Adoptium](https://adoptium.net/).

## Install

When installing Dpm, you can set the `DPM_HOME` environment variable to change the location where the SDK and any future updates are installed. The default is

- `${HOME}/.dpm/` on mac and linux
- `%APPDATA%/.dpm/` on windows

### Installation Instructions

To install the latest version:

#### Mac/Linux Installation

```bash
curl https://get.digitalasset.com/install/install.sh | sh
```

#### Windows Installation

Download and run the [windows installer](https://get.digitalasset.com/install/latest-windows.html), which will install the dpm sdk and set up the PATH variable for you.

### Manual Installation Instructions

If you prefer a more manual installation process, see `here <dpm-manual-installation>`.

### Managing and Upgrading SDK Versions

You can manage SDK versions manually by using `dpm install`.

To install the SDK version specified in the daml.yaml, run:

```bash
dpm install package
```

To install a specific SDK version, for example version `3.4.9`, run:

```bash
dpm install 3.4.9
```

To see the active SDK version

```bash
dpm version --active
```

```bash
3.4.0-snapshot.20251006.1451.85eca5a
```

To list the installed SDK versions, including the currently active one (marked with <span class="title-ref">\*</span>):

```bash
dpm version
```

```bash
3.4.0-snapshot.20251003.1412.3fe167f
* 3.4.0-snapshot.20251006.1451.85eca5a
```

To additionally list all the SDK versions that can be installed, as well as the installed versions:

```bash
dpm version --all
```

To get the list in a machine readable format:

```bash
dpm version --all -o json
```

```json
[
   {
       "version": "3.4.0-rc2",
       "remote": true
   },
   {
       "version": "3.4.8",
       "installed": true,
       "remote": true
   },
   {
       "version": "3.4.9",
       "installed": true,
       "remote": true,
       "active": true
   }
]
```

### Unstable releases

To install unstable SDKs you need to `configure dpm <dpm-configuration>` to look for them by setting the `registry` configuration field or `DPM_REGISTRY` environment variable to <span class="title-ref">europe-docker.pkg.dev/da-images/public-unstable</span>, then you can use the same `dpm install` command:

```bash
dpm install <unstable SDK version>
```

## Operate

- `dpm build`: Build a Daml package or project

  This builds the Daml project according to the project config file `daml.yaml` (see <span class="title-ref">configuration files \<dpm-configuration-files\></span>).

  In particular, it will use the dpm SDK (specified in the `sdk-version` field in `daml.yaml`) to resolve dependencies and compile the Daml project.

  Given a `daml.yaml` and `.daml` source files, the `dpm build` command will generate a .dar for this package. See `How to build Daml Archives <build_howto_build_dar_files>` for how to define a package and build it to a DAR.

- `dpm test`: Test the current Daml project or the given files by running all test declarations.

  This runs all daml scripts defined within a package.

  Daml Scripts are top level values of type `Script ()`, from the `daml-script` package. This package mimics a Canton Ledger Client for quick iterative testing, and direct support within `Daml Studio <daml-studio>`. The command runs these scripts against a reference Ledger called the IDE Ledger, which implements the core functionality of the Canton Ledger without the complexity of multi-participant setups.

  It is most useful for verifying the fundamentals of your ledger model, before moving onto integration testing via the Ledger API directly, or the Daml Codegen. `dpm test` also provides code coverage information for templates and choices used.

- `dpm clean`: Clean a Daml package or project

  This removes any Daml artifact files created in your package during a daml build, including DARs.

- `dpm codegen-alpha-java`: codegen (alpha) for java

- `dpm codegen-alpha-scala`: codegen (alpha) for scala

- `dpm codegen-alpha-typescript`: codegen (alpha) for typescript

- `dpm codegen-java`: Daml to Java compiler

- `dpm codegen-js`: Daml to JavaScript compiler

- `dpm daml-shell`: daml-shell client for PQS

- `dpm damlc`: Compiler and IDE backend for the Daml programming language

- `dpm docs`: Generate documentation for a daml package from its documentation comments

- `dpm inspect-dar`: Inspect a DAR archive

- `dpm new`: Create a new Daml package

- `dpm pqs`: participant query store

- `dpm sandbox`: Run full Canton installation in a single process

- `dpm script`: Daml Script Binary

- `dpm studio`: Launch Daml Studio

<!-- -->

- `dpm validate-dar` Validate a DAR archive

- `dpm install`: Install new SDK versions manually

  Note that you need to update your <span class="title-ref">project config file \<#configuration-files\></span> to use the new version.

| daml command                 | dpm command                                                                     | purpose                           |
|------------------------------|---------------------------------------------------------------------------------|-----------------------------------|
| daml new                     | dpm new                                                                         | Create a new Daml project         |
| daml build                   | dpm build                                                                       | Compile the Daml project          |
| daml test                    | dpm test                                                                        | Run tests for the Daml project    |
| daml install                 | dpm install                                                                     | Install Daml SDK components       |
| daml codegen java            | dpm codegen-java                                                                | Java code generation              |
| daml codegen js              | dpm codegen-js                                                                  | TypeScript code generation        |
| daml damlc                   | dpm damlc                                                                       | Invoke the daml compiler          |
| daml studio                  | dpm studio                                                                      | Open project in Visual Studio     |
| daml sandbox                 | dpm sandbox                                                                     | Launch a Daml Sandbox             |
| daml ledger allocate-parties | Use Declarative API – OR – JSON / gRPC API                                      | Allocate parties on a ledger      |
| daml ledger list-parties     | JSON / gRPC API                                                                 | list parties on a ledger          |
| daml ledger upload-dar       | Use Declarative API – OR – JSON / gRPC API                                      | Upload (and vet) dars on a ledger |
| daml ledger fetch-dar        | gRPC API                                                                        | Fetch a Dar from a ledger.        |

## Key commands

Once installed, `dpm` is your primary interface to the SDK. Here are the commands you'll use most often.

**Project scaffolding and building:**

```bash
# Create a new Daml project
dpm new

# Compile Daml source into a DAR file
dpm build

# Run all test scripts in the project
dpm test

# Remove build artifacts
dpm clean
```

**Code generation:**

```bash
# Generate TypeScript/JavaScript bindings from a DAR
dpm codegen-js .dar -o generated

# Generate Java bindings from a DAR
dpm codegen-java .dar -o generated
```

**Local development:**

```bash
# Start a local Canton sandbox node
dpm sandbox

# Launch Daml Studio in VS Code
dpm studio

# Run a Daml Script against a ledger
dpm script
```

**Other useful commands:**

```bash
# Inspect the contents of a DAR archive
dpm inspect-dar

# Validate a DAR archive
dpm validate-dar

# Generate documentation from Daml doc comments
dpm docs
```

Run any command with `--help` to see its full options.

## SDK version management

To use unstable (pre-release) SDK versions, set the `DPM_REGISTRY` environment variable to `europe-docker.pkg.dev/da-images/public-unstable` before running `dpm install`.

## Project configuration

Each Daml project has a `daml.yaml` file that defines the package name, SDK version, source directory, and dependencies. A multi-package project uses an additional `multi-package.yaml` to list the directories of connected packages so that `dpm build` can resolve them in the correct order.

Both config files support environment variable interpolation (`${MY_VAR}` syntax), which is useful for extracting common values like the SDK version or dependency paths into `.envrc` files or build systems.

For full configuration details, see the [DPM configuration reference](https://docs.digitalasset.com/build/3.5/dpm/configuration.html).

## Migrating from the `daml` assistant

If you previously used the `daml` CLI, `dpm` is a drop-in replacement. The mapping is straightforward:

- `daml build` becomes `dpm build`
- `daml test` becomes `dpm test`
- `daml codegen java` becomes `dpm codegen-java`
- `daml codegen js` becomes `dpm codegen-js`
- `daml sandbox` becomes `dpm sandbox`
- `daml studio` becomes `dpm studio`
- `daml new` becomes `dpm new`

Commands that previously used `daml ledger` (such as `allocate-parties`, `upload-dar`, and `list-parties`) now use the Declarative API or the JSON/gRPC API directly.
