### Readiness and Liveness Probes

All Canton Network applications provide `/readyz` and `/livez` endpoints for readiness and liveness probes.

**Kubernetes** — Probes are pre-configured. You can also manually check readiness:

```bash
kubectl exec <pod-name> -n <namespace> -- curl -v https://localhost:5003/api/validator/readyz
```

**Docker** — Check liveness inside the container:

```bash
docker exec <container-name> -- curl -v https://localhost:5003/api/validator/livez
```

HTTP status 200 indicates the validator is ready and live.

### Last Ingested Record Time

The `splice_store_last_ingested_record_time_ms` metric tracks the last ingested record time in each validator store. Use it to verify your node is keeping up with the network:

- **Value increasing over time** — Your node is active and in sync. For validators collecting liveness rewards, this advances every round, so lag should never exceed 20 minutes.
- **Value static** — Your node may be stalled. Investigate further.

The dedicated Grafana dashboard **Splice Store Last Ingested Record Time** visualizes this metric.

## Critical Metrics by Category

### Synchronizer Connectivity

| Metric | What It Tells You |
|--------|-------------------|
| Sequencer connection status | Whether your participant can reach the Global Synchronizer |
| Reconnection attempts | Frequency of connection drops (network instability indicator) |
| Message processing latency | How quickly your node processes sequenced messages |

A sustained loss of sequencer connectivity means your node can't submit or confirm transactions.

### Transaction Processing

| Metric | What It Tells You |
|--------|-------------------|
| Command submission rate | Volume of transactions your node is submitting |
| Command completion latency | End-to-end time from submission to completion |
| Rejection rate | Proportion of submitted commands that fail validation |

High rejection rates may indicate package vetting issues, incorrect party hosting, or application bugs. Monitor completion latency to detect synchronizer congestion or participant overload.

### Database Health

| Metric | What It Tells You |
|--------|-------------------|
| Database connection pool utilization | Whether your pool is sized correctly |
| Query latency (p50, p99) | Database performance under load |
| Storage growth rate | How fast your database is growing |

Database performance directly affects transaction processing speed. If connection pool utilization is consistently above 80%, increase the pool size or scale your database.

### Traffic and Canton Coin

| Metric | What It Tells You |
|--------|-------------------|
| Traffic balance | Remaining traffic credit on the synchronizer |
| Traffic consumption rate | How fast your node is consuming traffic |

A zero traffic balance prevents your node from submitting transactions. Monitor consumption rate to plan traffic purchases.

## Alerting Recommendations

Set up alerts for these conditions:

- **Node not ready** — `/readyz` returns non-200 for more than 5 minutes
- **Ingestion stalled** — `splice_store_last_ingested_record_time_ms` hasn't advanced in 30 minutes
- **Sequencer disconnected** — No active sequencer connection for more than 2 minutes
- **Database connection pool exhausted** — Pool utilization above 95%
- **Traffic balance low** — Below the amount needed for 24 hours of normal operation
- **Disk usage high** — Database storage above 80% capacity
